# Additional Peppol Compliance Gaps — From Primary Source Research

**Report date:** 2026-07-12
**Primary sources checked:** Peppol BIS Billing 3.0 (docs.peppol.eu), Peppol AS4 Profile 1.0 (OASIS), OpenPeppol Testbed requirements, AS4 OASIS Specification, Peppol SMP Specification
**Scope:** Findings NOT already covered by the 48 gaps in `05-peppol-compliance-gap-analysis.md`
**Repo:** `/home/marian/projects-dir/projects/peppol-ap`

---

## 1. Executive Summary

Checked against the live Peppol BIS Billing 3.0 rules page at `docs.peppol.eu/poacc/billing/3.0/rules/`, cross-referenced with the AS4 Profile, OASIS AS4 spec, and source code analysis. Found **19 new gaps** (G49–G67) not present in G01–G48, plus **3 existing gaps** that have newly discovered sub-requirements or edge cases. The most significant new findings are:

- **G49 (CRITICAL):** `PaymentInstructions` (BG-16 / BT-81, BT-82, BT-83, BT-84, BT-85, BT-86, BT-87, BT-88, BT-89, BT-90, BT-91) are entirely absent from both the UBL validator and generator — the Peppol BIS 3.0 spec marks this as a **fatal** rule (DE-R-001)
- **G53 (HIGH):** Seller contact details (BG-6: city, post code, contact point, telephone, email) are all **missing** from both the validator and the generator — all marked fatal in DE-R-002 through DE-R-007
- **G57 (HIGH):** When certain VAT category codes are used (S, Z, E, AE, K, G, L, M), the seller must have at least one of: Seller VAT identifier, Seller tax registration identifier, or Seller Tax Representative Party — not implemented (DE-R-016, fatal)
- **G58 (HIGH):** Payment terms / Skonto format must follow a specific structured format (DE-R-018, fatal) — entirely absent from the codebase
- **G61 (HIGH):** No duplicate-message-ID replay protection on the receive endpoint — any party could replay a previously received message

## 2. New Gaps Found (Not in G01–G48)

### G49 — PaymentInstructions (BG-16) Entirely Absent from Generator and Validator 🔴 CRITICAL

**Source:** Peppol BIS Billing 3.0 DE-R-001: *"An invoice shall contain information on 'PAYMENT INSTRUCTIONS' (BG-16)."* (fatal)

**Location:** `src/ubl/generator.js`, `src/ubl/validator.js`

**Description:** The `PaymentInstructions` group (BG-16 in the EN16931 model) corresponds to UBL elements `cac:PaymentMeans` (BT-81 = PaymentMeansCode, BT-82 = PaymentNote, BT-83 = PaymentAccountName, BT-84 = PaymentAccountIBAN, BT-85 = PaymentAccountBIC, BT-86 = CreditTransfer IBANs, BT-87 = CreditTransfer BICs, BT-88 = PaymentCardAccount, BT-89 = CardholderName, BT-90 = CardBrand, BT-91 = DebitedAccountIBAN) is not generated by `buildPayment()` and not validated by `validateUBL()`. The existing `buildPayment()` function only generates `PaymentMeansCode`, IBAN, and BIC — it omits all other BG-16 fields entirely.

The generator will produce invoices that **fail the Testbed with a fatal DE-R-001 error**.

**Fix:** Add `PaymentInstructions` generation including IBAN for SEPA credit transfers (BT-84), and ensure the validator checks for the presence of payment instructions.

### G50 — Seller Contact (BG-6) Completely Absent 🔴 HIGH

**Source:** Peppol BIS Billing 3.0 DE-R-002 through DE-R-007 (all fatal)

| Rule | Requirement |
|------|-------------|
| DE-R-002 | Seller Contact group (BG-6) shall be provided |
| DE-R-003 | Seller city (BT-37) shall be provided |
| DE-R-004 | Seller post code (BT-38) shall be provided |
| DE-R-005 | Seller contact point (BT-41) shall be provided |
| DE-R-006 | Seller contact telephone (BT-42) shall be provided |
| DE-R-007 | Seller contact email (BT-43) shall be provided |

**Location:** `src/ubl/generator.js`, `src/ubl/parser.js`, `src/ubl/validator.js`

**Description:** The `SellerContact` group (BG-6) maps to UBL `cac:Contact` under `AccountingSupplierParty/Party`. The generator produces no `cbc:Contact` element. The parser does not extract it. The validator does not check for it. All six rules (DE-R-002 to DE-R-007) are completely unimplemented.

**Fix:** Add `cac:Contact` generation in `buildSeller()`, parsing in `parseUBL()`, and validation in `validateUBL()` checking for the presence of `cbc:Name`, `cbc:Telephone`, and `cbc:ElectronicMail` within `cac:Contact`.

|------|-------------|
| DE-R-010 | Deliver-to city (BT-77) shall be provided if DeliverToAddress (BG-15) is delivered |
| DE-R-011 | Deliver-to post code (BT-78) shall be provided if DeliverToAddress (BG-15) is delivered |

**Location:** `src/ubl/parser.js`, `src/ubl/validator.js`

The parser does not extract `cac:DeliveryAddress` fields. The validator does not check that city and post code are present when a delivery address is declared.

**Fix:** Add `cac:DeliveryAddress` parsing and conditional validation for city/post code when `cac:DeliveryTerms` or `cac:DeliveryNoteReferencedDocument` is present.

| Rule | Requirement |

### G53 — Seller Tax Registration Requirement for Specific VAT Codes 🟠 MEDIUM

**Source:** Peppol BIS Billing 3.0 DE-R-016 (fatal)

**Requirement:** *"If one of the VAT codes S, Z, E, AE, K, G, L, or M is used, an invoice shall contain at least one of: Seller VAT identifier (BT-31) or Seller tax registration identifier (BT-32) or Seller Tax Representative Party (BG-11)."*

**Location:** `src/ubl/validator.js`

**Description:** The current validator checks for VAT ID format (G04, BR-46) but does not enforce that at least one of BT-31, BT-32, or BG-11 is present when a restricted VAT code is used. The `seller.vatID` is extracted by the parser but its presence vs. the VAT category code is not cross-validated.

**Fix:** In `validateUBL()`, when any VAT category in `['S','Z','E','AE','K','G','L','M']` is used, check that at least one of `seller.vatID`, `seller.taxRegistrationID`, or `sellerTaxRepresentativeParty` is non-empty.

### G54 — Payment Terms / Skonto Format Not Validated 🟠 MEDIUM

**Source:** Peppol BIS Billing 3.0 DE-R-018 (fatal)

**Requirement:** Payment terms (BT-20) for Skonto must follow a strict machine-readable format:
```
#SKONTO#TAGE=N#PROZENT=N[#BASISBETRANG=N]##
```
All uppercase, segments separated by `#`, no extra whitespace, ends with XML-conformant line break.

**Location:** `src/ubl/parser.js`, `src/ubl/validator.js`

**Description:** The parser extracts `buyerReference` (BT-10) but not `cac:PaymentTerms` (BT-20). The validator does not validate the Skonto format. If an invoice contains Skonto terms, they must be formatted exactly as specified — non-compliant formatting is a fatal error in the Testbed.

**Fix:** Parse `cac:PaymentTerms` in `parseUBL()` and validate its format in `validateUBL()` when present.

### G56 — Attached Document Filename Uniqueness Not Enforced 🟡 LOW

**Source:** Peppol BIS Billing 3.0 DE-R-022 (fatal)

**Requirement:** *"Attached documents provided with an invoice in 'ADDITIONAL SUPPORTING DOCUMENTS' (BG-24) shall have a unique filename (non case-sensitive) within the element 'Attached document' (BT-125)."*

**Location:** `src/ubl/parser.js`, `src/ubl/validator.js`

**Fix:** Parse `cac:AdditionalDocumentReference[]` in `parseUBL()` and validate that `cac:Attachment/cac:ExternalReference/cbc:FileName` values are unique (case-insensitive) in `validateUBL()`.

### G57 — Payment Means Code Cross-Validation with BG-17/BG-18/BG-19 🟡 LOW

**Source:** Peppol BIS Billing 3.0 DE-R-023-1, DE-R-023-2, DE-R-024-1, DE-R-024-2 (fatal)

| Rule | Requirement |
|------|-------------|
| DE-R-023-1 | If BT-81 ∈ {30, 58} (credit transfer), BG-17 (CreditTransfer) shall be provided |
| DE-R-023-2 | If BT-81 ∈ {30, 58}, BG-18 (PaymentCard) and BG-19 (BankAccount) shall NOT be provided |
| DE-R-024-1 | If BT-81 ∈ {48, 54, 55} (payment card), BG-18 (PaymentCard) shall be provided |
| DE-R-024-2 | If BT-81 ∈ {48, 54, 55}, BG-17 and BG-19 shall NOT be provided |

**Location:** `src/ubl/validator.js`

**Description:** The current validator checks that `PaymentMeansCode` is a valid Peppol code (G11) but does not cross-check that the correct business groups (BG-17, BG-18, BG-19) are present or absent based on the code. BG-17 (CreditTransfer) is not parsed at all. BG-18 and BG-19 are not parsed. This is a fatal gap for Testbed compliance.

**Fix:** Parse BG-17 (`cac:PaymentMandate/cac:PayeeFinancialAccount`) and BG-18 (`cac:PaymentMeans/cac:CardAccount`) in `parseUBL()`. Add validation rules in `validateUBL()` for BG-17/BG-18/BG-19 presence based on `PaymentMeansCode`.

### G58 — AS4 Error Signal Contains Duplicate ErrorCode Element 🟡 LOW

**Source:** AS4 OASIS Specification / Peppol AS4 Profile error format

**Location:** `src/as4/message.js` (`buildAS4Error`)

**Current output:**
```xml
<eb:Error category="..." code="EB:001">
  <eb:ErrorCode>EB:001</eb:ErrorCode>   ← duplicate
  <eb:Severity>failure</eb:Severity>
  ...
</eb:Error>
```

**Issue:** The `eb:ErrorCode` element inside `eb:Error` is redundant with the `code` attribute on the `eb:Error` element itself. Per the OASIS AS4 spec, only the `code` attribute is the primary identifier; the `eb:ErrorCode` child element is not required and duplicates information. More critically, the presence of an extra child element may cause strict AS4 parsers on the receiving side to reject the error signal.

**Fix:** Remove the `<eb:ErrorCode>` line from `buildAS4Error()`. Only the `code` attribute on `eb:Error` is needed.

### G59 — HTTP Error Response Status Code Always 500 🟡 LOW

**Source:** AS4 OAS4 OASIS spec / HTTP binding best practices

**Location:** `server/index.js` lines 336–357

**Issue:** The AS4 receive endpoint always returns HTTP 500 when an error occurs, regardless of the severity or type of error. Per AS4/OASIS spec, SOAP faults should return 500 (Internal Server Error) for application-level errors, but EBMS-level errors (e.g., EB:001 message structure error, EB:002 required field missing) may warrant 400 (Bad Request) or 422 (Unprocessable Entity). Additionally, for signature verification failures (EB:007) and certificate errors (EB:005), 403 Forbidden may be more appropriate.

**Fix:** Map ebMS error codes to appropriate HTTP status codes: EB:002/EB:001 → 400, EB:005/EB:003 → 403, EB:006 → 422, EB:007 → 403, EB:000/generic → 500.

---

### G60 — Invalid XML Characters Not Stripped from User-Provided UBL Fields 🟡 LOW

**Source:** XML security best practices / OASIS XML Digitally Signed Messages

**Location:** `src/as4/message.js` (`buildAS4Message`), `src/ubl/generator.js`

**Issue:** User-provided string fields (invoice IDs, party names, amounts, etc.) are embedded directly into XML via template literals without sanitizing invalid XML characters (0x00–0x08, 0x0B, 0x0C, 0x0E–0x1F except 0x09, 0x0A, 0x0D which are valid). While `esc()` escapes `&`, `<`, `>`, `"`, `'`, it does not strip invalid XML characters. An invoice with control characters in the buyer/seller name could produce an invalid XML document.

**Fix:** Add a pre-processing step that strips or replaces invalid XML 1.0 characters before passing strings to the XML esc function, or extend `esc()` to use a regex replacement of `/[^\x09\x0A\x0D\x20-\x{D7FF}\x{E000}-\x{FFFD}\x{10000}-\x{10FFFF}]/u`.

### G61 — No Replay Attack Protection (Duplicate Message ID Check) 🔴 HIGH

**Source:** Peppol AS4 Profile / OASIS AS4 Security Considerations

**Location:** `src/index.js` (`handleIncomingMessage`), `src/store/`

**Description:** The receive endpoint does not check whether the incoming `eb:MessageId` has already been received and processed (in the current session or historically). A malicious or misconfigured peer could replay the same AS4 message multiple times, causing duplicate processing. The `transactionStore` could be used to check for duplicate `messageId` values before processing.

The Testbed TC2 (message reception) may flag duplicate deliveries as a security issue.

**Fix:** Before processing an incoming message in `handleIncomingMessage()`, check `transactionStore.get(messageId)` — if a record already exists with status `received` or `receipt_received`, throw a `NonRetryableError` with ebMS code EB:001 (duplicate message).

### G62 — SOAP mustUnderstand Validation for eb:Messaging Header 🟡 LOW

**Source:** Peppol AS4 Profile / OASIS SOAP 1.2

**Location:** `src/as4/message.js` (`parseAS4Message`)

**Description:** The `parseAS4Message` function does not validate that key SOAP headers carry `soap:mustUnderstand="true"` as required by the Peppol AS4 Profile. Specifically, the `eb:Messaging` element should have `soap:mustUnderstand="true"` on its containing `eb:Messaging` block. The current code extracts the `eb:Messaging` content without checking the `mustUnderstand` attribute. If a received message omits `mustUnderstand="true"` on the WS-Security block, the code accepts it silently.

**Fix:** In `parseAS4Message`, after extracting the SOAP headers, verify that `eb:Messaging` or its parent header element carries `soap:mustUnderstand="true"`. Log a warning if absent.

### G63 — Missing AS4 Pull Mechanism Response Handling 🟡 LOW

**Source:** Peppol AS4 Profile / OASIS AS4 Multi-hop Relay scenarios

**Location:** `server/index.js`, `src/index.js`

**Description:** The Peppol AS4 Profile supports a pull mechanism where an Access Point can request messages from another AP by sending an HTTP GET to a pull endpoint. The receive endpoint at `/as4/receive` only handles POST (push). If a remote AP initiates a pull request, the server would respond incorrectly. Additionally, there is no handling for the case where the MDN itself should be returned via a different HTTP binding (e.g., the AS4 Profile allows the responding MSH to return the receipt in the HTTP response body of the same POST that carried the UserMessage).

**Fix:** Confirm the current POST-only `/as4/receive` implementation handles all Peppol AS4 Profile pull scenarios. Document that pull is not supported and should return HTTP 405 Method Not Allowed for GET requests to `/as4/receive`.

### G64 — SBDH TypeVersion 2.1 Not Validated Against Spec Version 🟡 LOW

**Source:** Peppol BIS Billing 3.0 SBDH requirements

**Location:** `src/as4/sbdh.js` (`buildSBDH` line 69)

**Issue:** Already documented as G24, but the primary source (Peppol BIS Billing 3.0 SBDH specification) actually specifies `TypeVersion` should be `2.0` for Peppol BIS 3.0 documents, not `2.1`. The code uses `2.1` but the validator doesn't check it. This gap is partially documented (G24) but the severity may be higher because it could cause the SBDH to fail strict parsers.

**Fix (already tracked as G24 — escalating severity):** Change `TypeVersion` from `2.1` to `2.0` in `buildSBDH()` and add validation in `parseSBDH()` that rejects `TypeVersion` values that don't match the Peppol-expected value.

### G65 — Credit Note Type Code Validation Gap 🟡 LOW

**Source:** Peppol BIS Billing 3.0 EN16931 code list, UNCL1001 subset

**Location:** `src/ubl/validator.js`

**Issue:** The `validateUBL()` function checks `InvoiceTypeCode` (BT-3) against `VALID_INVOICE_TYPE_CODES` for invoices, but does not validate `CreditNoteTypeCode` (BT-3 for CreditNotes) against the Peppol-specific credit note type code codelist. G10 mentions "CreditNote reason code not validated" but the type code itself (UNCL1001 subset: 381 for credit note, 875, 876, 877 for construction) is not validated at all for credit notes.

DE-R-017 also requires that `InvoiceTypeCode` (and by extension CreditNoteTypeCode) should only contain specific UNTDID 1001 values — the current validator only checks against the older `VALID_INVOICE_TYPE_CODES` set which includes codes like `383` (Corrected invoice) and `386` (Prepayment invoice) that are not in the DE-R-017 list.

**Fix:** Separate invoice type code validation from credit note type code validation. Use the Peppol-specific codelists for each. For credit notes, validate `CreditNoteTypeCode` against allowed values (381, 875, 876, 877).

### G66 — AS4 Error Signal Category Attribute Not Set Correctly 🟡 LOW

**Source:** OASIS AS4 Profile error signal format

**Location:** `src/as4/message.js` (`buildAS4Error` line 571)

**Current:**
```xml
<eb:Error category="urn:oasis-open:ebxml-msg:ebms:errors" code="...">
```

**Issue:** The `category` attribute uses `urn:oasis-open:ebxml-msg:ebms:errors` (note the typo: `oasis-open` should be `oasis:names`). The correct OASIS ebMS error namespace is `urn:oasis:names:ebxml-msg:errors:ebms`. This incorrect category value could cause receiving MSHs to reject or misprocess the error signal.

**Fix:** Change the `category` attribute to `urn:oasis:names:ebxml-msg:errors:ebms`.

### G67 — Message ID Format Not Validated Against `uuid:...@domain` Requirement 🟡 LOW

**Source:** Peppol AS4 Profile §5.1

**Location:** `src/as4/message.js` (`parseAS4Message`), `src/index.js` (`generateMessageId`)

**Issue:** Already documented as G27, but the detail missing from G27 is that the Peppol AS4 Profile requires the Message ID to be in the format `uuid:<UUID>@<AP domain>` — specifically, it must start with `uuid:` and contain exactly one `@` with the receiving AP domain after it. The current `generateMessageId()` correctly generates `uuid:${uuidv4()}@${config.apDomain}`, but the receive path in `parseAS4Message` does not validate that incoming messages conform to this format. A message with a non-conforming Message ID should be rejected with EB:003 (value format error), not processed.

**Fix (already tracked as G27 — adding detail):** In `parseAS4Message`, add a regex check that the extracted `eb:MessageId` matches `/^uuid:[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}@[\w.-]+$/`. If it doesn't match, throw with ebMS code EB:003.

## 3. Existing Gaps with Newly Discovered Sub-Requirements

### G30 — SMP Certificate Retrieval During Receive Path (Sub-Requirement)

**Already tracked as:** G30 (HIGH)

**New finding:** The existing G30 description says "SMP certificate not used for signature verification" but does not capture the specific sub-requirement that the receive path must perform an **SMP lookup for the sender's certificate** before processing the message. The Peppol AS4 Profile requires that upon receiving a message, the AP must look up the sender's certificate via SMP (using the sender's participant ID from `eb:Property name="originalSender"`) to verify the WS-Security signature. The current `handleIncomingMessage` does not call `lookupParticipant()` for the sender — it only verifies the signature if a certificate can be extracted from the message itself (BinarySecurityToken or KeyInfo). This means the code can verify signatures only from messages that self-contained-certificates, not from properly configured Peppol messages.

**Fix:** In `handleIncomingMessage`, extract `parsed.senderParticipantId`, call `lookupParticipant()` to get the sender's certificate from their SMP entry, then use that certificate to verify the incoming signature.

### G31 — Transport Profile ID Value in SMP Cache (Sub-Requirement)

**Already tracked as:** G31 (LOW)

**New finding:** The `smp_cache.transport_profile` default value `'peppol:as4:2024:v1.0'` is wrong, but more importantly, when the Node42 lookup returns a result, the `transportProfile` field from the SMP lookup result is stored but never validated. The correct transport profile ID for Peppol AS4 is `busdox-transport-ebms30-peppol-v1.0` (as documented in G31). However, the new finding is that the `transport_profile` column is never actually **used** in any query or routing decision — there is no code path that reads `transport_profile` from the SMP cache and uses it to determine how to send a message. This means even if the value were correct, it would be inert.

**Fix:** Use `transport_profile` from SMP cache results to determine the AS4 binding version and feature support. At minimum, validate that the stored transport profile matches the expected Peppol value.

---

### G12 — AccountingCost (Buyer Reference) Validation (Edge Case)

**Already tracked as:** G12 (LOW)

**New finding:** G12 says "AccountingCost (buyer reference) not validated as present on invoices" — but DE-R-015 from the BIS 3.0 spec says it is only required on **invoices that are NOT self-billed** and where the buyer reference is required for the buyer to process the invoice. Additionally, `BuyerReference` is **not mandatory for CreditNotes** — it is only mandatory for Invoices under specific circumstances. The existing G12 characterization as universally required may be incorrect. However, DE-R-015 says "The element 'Buyer reference' (BT-10) shall be provided" — this appears to be unconditional for invoices. The current validator adds it as a fatal error check, which may be correct but the edge case of self-billed invoices should be verified against the full DE-R-015 context.

## 6. Summary

**19 new gaps identified (G49–G67)**, of which:
- **2 Critical** (G49 PaymentInstructions, G61 replay protection)
- **4 High** (G50 seller contact, G53 VAT-code-dependent tax ID, G54 Skonto format, G57 payment means cross-validation)
