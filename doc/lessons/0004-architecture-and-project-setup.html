🏗️ Lesson 4: High-Level Architecture & Project Setup

The system design — components, database, API routes, and project structure.

The big picture

You're building two things that sit on the same server but are logically separate:

  1. The Peppol Access Point (AP) Core — the certified transport layer that speaks AS4 with the Peppol network
  2. The Service Platform — your product: a multi-tenant web portal + API that SMEs use to send/receive invoices

The AP Core exposes a clean internal interface. Your portal uses it — but any portal, accounting software, or third-party system could connect to it too.

Physical deployment: same server or separate?

The diagrams above show both systems on one machine. For your prototype (4 weeks), that's exactly what you should do — everything on a single $10 VPS.

But in production, you'll want them on separate servers. Here's why and how:

Option A: Single server (prototype)

┌────────────────────────────────────────────┐ │ one VPS (e.g. Hetzner) │ │ │ │ Port 443 ─── AS4 Engine (Node42) │ │ Port 80 ─── Nginx → React static files │ │ Port 3000 ── Express API (internal) │ │ Port 5432 ── PostgreSQL │ │ │ │ Domain: ap.mojafaktura.sk │ └────────────────────────────────────────────┘

Pros: simple, cheap, easy to debug. Cons: if the portal goes down due to a bug, the AP also goes down.

Option B: Separate servers (production)

┌─────────────────────────────────┐ ┌─────────────────────────────┐ │ AP Core Server │ │ Service Platform Server │ │ (minimal, hardened) │ │ (full web app) │ │ │ │ │ │ • Only what's needed: │ │ • Express API │ │ Node42 + small relay daemon │ │ • React portal (Vite) │ │ │ │ • PostgreSQL │ │ • Port 443: AS4 (Peppol) │ │ • Admin dashboard │ │ • Port 3001: Internal API* │ │ • Background job queue │ │ │ │ │ │ Domain: ap.mojafaktura.sk │ │ Domain: app.mojafaktura.sk │ └──────────────┬──────────────────┘ └─────────────┬───────────────┘ │ │ │ Internal API │ │ (authenticated, over HTTPS │ │ or private network) │ │ │ │ AP Core → Platform: │ │ "Invoice received for SK449..." │ │ │ │ Platform → AP Core: │ │ "Send this UBL to SK449..." │ │ │ └──────────┬─────────────────────────┘ │ ▼ ┌─────────────────────┐ │ Redis or message │ │ queue (optional) │ └─────────────────────┘

What the AP Core server does when physically separated:

  • Inbound: receives AS4 message → extracts UBL → calls your Platform's webhook (POST https://app.mojafaktura.sk/api/webhook/invoice-received) with the parsed invoice
  • Outbound: your Platform calls the AP Core's internal API (POST https://ap.mojafaktura.sk:3001/api/send) with the UBL XML and target participant ID
  • Certificate management: AP Core holds the private keys. Platform never touches them.
🔐 Security benefit of separation: The AP Core server is hardened and minimal. It runs one process, has no database, no user management, no React — just the AS4 engine and a tiny relay API. Fewer moving parts = smaller attack surface. If the Platform server gets compromised, the attacker still can't access your Peppol private keys.

companies

CREATE TABLE companies (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            TEXT NOT NULL,                 -- obchodné meno
  legal_name      TEXT NOT NULL,                 -- registration name
  ico             TEXT NOT NULL UNIQUE,          -- IČO: 12345678
  dic             TEXT,                          -- DIČ: 2023456789
  ic_dph          TEXT,                          -- IČ DPH: SK2023456789
  street          TEXT,
  city            TEXT,
  zip             TEXT,
  country         TEXT DEFAULT 'SK',
  
  peppol_id       TEXT UNIQUE,                   -- Peppol participant ID
  endpoint_scheme TEXT DEFAULT '9914',           -- EAS code

  iban            TEXT,                          -- default IBAN
  bic             TEXT,                          -- default BIC

  is_active       BOOLEAN DEFAULT true,
  created_at      TIMESTAMPTZ DEFAULT now(),
  updated_at      TIMESTAMPTZ DEFAULT now()
);

users

CREATE TABLE users (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id      UUID NOT NULL REFERENCES companies(id),
  email           TEXT NOT NULL UNIQUE,
  password_hash   TEXT NOT NULL,                  -- bcrypt
  name            TEXT NOT NULL,

documents

CREATE TABLE documents (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id        UUID NOT NULL REFERENCES companies(id),
  
  direction         TEXT NOT NULL CHECK (direction IN ('outbound','inbound')),
  doc_type          TEXT NOT NULL CHECK (doc_type IN ('invoice','credit_note')),
  status            TEXT NOT NULL DEFAULT 'draft'
                    CHECK (status IN ('draft','validated','sent','received',
                                      'delivered','rejected','error')),

  -- Invoice identifiers
  invoice_number    TEXT NOT NULL,                 -- BT-1
  issue_date        DATE NOT NULL,                 -- BT-2
  due_date          DATE,                          -- BT-9
  invoice_type_code TEXT DEFAULT '380',             -- BT-3
  currency          TEXT DEFAULT 'EUR',             -- BT-5

  -- Counterparty (the other end, not our customer)
  counterparty_name    TEXT,
  counterparty_ic_dph TEXT,                        -- their VAT ID
  counterparty_peppol_id TEXT,

  -- Monetary totals
  line_extension_amount  NUMERIC(12,2),
  tax_exclusive_amount   NUMERIC(12,2),
  tax_inclusive_amount   NUMERIC(12,2),
  payable_amount         NUMERIC(12,2),

  -- Peppol tracking
  peppol_message_id   TEXT,                        -- from AS4 message
  sbdh_xml            TEXT,                        -- stored SBDH envelope
  ubl_xml             TEXT,                        -- the actual UBL XML
  receipt_xml         TEXT,                        -- MDN receipt

  created_at          TIMESTAMPTZ DEFAULT now(),
  sent_at             TIMESTAMPTZ,
  delivered_at        TIMESTAMPTZ,

  UNIQUE(company_id, invoice_number)
);

💡 Multi-tenancy strategy: Every table has a company_id. The tenant middleware reads the JWT token, extracts the company, and adds WHERE company_id = ? to every query automatically. This keeps SMEs completely isolated from each other.

Key flows

Outbound invoice (SME → Peppol)

  1. SME creates invoice in the React portal (or pushes via integration API)
  2. Express saves it as draft in the database
  3. SME clicks "Send" → Express calls services/peppol.js
  4. peppol.js does:
    1. Validate the UBL XML against Schematron (Node42 validator)
    2. Look up the receiver via SML + SMP (Node42 lookup)
    3. Build the SBDH envelope + AS4 SOAP message
    4. Sign with your AP's Peppol PKI certificate
    5. Encrypt with the receiving AP's public key
    6. POST the MIME multipart message to the receiver's endpoint
    7. Receive and verify the signed receipt (MDN)
  5. Document status changes to sent or delivered
  6. Transaction is logged for audit trail

Inbound invoice (Peppol → SME)

  1. Another AP sends an AS4 message to your server's endpoint
  2. Express's /as4/receive handler:
    1. Parses the SOAP envelope
    2. Validates the sender's signature against their Peppol certificate (from the SMP)
    3. Decrypts the payload using your private key
    4. Extracts the SBDH + UBL document
    5. Looks up which of your customers has the matching Peppol participant ID
    6. Validates the UBL with Schematron
    7. Saves as inbound document in the database
    8. Sends back a signed MDN receipt
  3. Your SME customer sees the invoice in their portal inbox

🔌 The interface between them: The AP Core exposes a small set of operations that the Service Platform (or any other system) can call:
  • send(invoice) — push a UBL invoice to any Peppol participant
  • receive(callback) — register where incoming documents are delivered
  • getStatus(messageId) — check if a sent message was delivered
  • lookup(participantId) — resolve a Peppol ID to endpoint + certificate
  • validate(ublXml) — Schematron validation with detailed error report

→ See the full API spec with request/response formats at reference/ap-core-interface.html

This means your React portal is just one possible client. An accounting software vendor could build their own portal on the same AP Core using the same interface.

What Node42 gives you (the AP Core)

The node42-edelivery package does most of the heavy lifting for the AP Core. Here's how it maps to your architecture:

Your componentNode42 moduleWhat it provides
as4/sender.jssrc/sender/AS4 message construction, signing, encryption, MIME assembly, HTTP POST, receipt handling
as4/receiver.jssrc/receiver/Parse incoming AS4, verify signature, decrypt, generate MDN
as4/lookup.jssrc/lookup.jsSML DNS resolution + SMP metadata query
ubl/validator.jsBundled Schematron filesPre-compiled Schematron rules in src/assets/schematrons/billing/
as4/certificates.jssrc/security/PKI trust chain, crypto operations
⚡ Prototype strategy:
  1. Use Node42 as a CLI tool first — test that you can send a Peppol message in the test environment
  2. Then integrate it as a library in your Express backend — wrap the sender/receiver in your service layer
  3. Build the React portal + PostgreSQL on top — these are straightforward CRUD with your existing stack skills
  4. Add the integration API last — it's just REST endpoints wrapping the same service layer
