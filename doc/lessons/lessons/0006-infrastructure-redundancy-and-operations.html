1. What the Peppol TIA Formally Requires

The Transport Infrastructure Agreement (TIA) Annex 3 sets the minimum SLA for all Peppol APs worldwide. These are the contractual minimums — you must meet them to stay accredited:

MetricRequirementWhat it means in practice
Business hours (Mon–Fri 9:00–16:00 CET) ≥ 98.5% available ~10.9 hours of downtime allowed per month
Outside business hours ≥ 94.0% available ~43.2 hours allowed per month
Unavailability definition Not reachable for ≥ 60 continuous seconds Short blips under 60 s don't count as downtime
Retry policy Sending AP: ≥ 3 retries within 2 hours Built-in protocol resilience gives you a 2-hour window
Data loss limit Max 24 hours of data loss After a disaster, you can lose up to a day of message metadata
Major incident notification Within 4 hours to Peppol Authority Security breaches and extended outages
Service windows Max 2 hours, outside business hours, 3 days notice Planned maintenance is accounted for
Capacity planning Documented process required Must show you track load and scale accordingly

Some Peppol Authorities impose stricter requirements. For example, the Norwegian financial sector requires 99.5% availability 24/7 — that's a much tighter bar (~3.6 hours/month total downtime).

💡 Key insight: The TIA minimum is surprisingly low. A single server with a documented backup/DR plan technically meets it. But commercial reality is different — customers expect much better, and your reputation (and accreditation renewal) depends on it.

2. What the Peppol Protocol Gives You (The Retry Safety Net)

The Peppol AS4 protocol itself provides significant operational slack, which directly affects your redundancy decisions:

  • Mandatory retries: The sending AP must retry delivery ≥3 times within 2 hours. If your AP goes down for 30 minutes, messages won't be lost — they'll queue at the sender.
  • Synchronous Receipt: AS4 is not fire-and-forget. The receiver must send back a signed <Receipt> signal. Until that receipt is received and verified, the sender considers the message undelivered and will retry.
  • Message-level idempotency: Every message has a unique eb:MessageId. If a receipt is lost and the sender retries, the receiver detects the duplicate and re-acknowledges without re-processing.
  • No real-time requirement: Peppol does not mandate real-time or near-real-time delivery. A message delivered within minutes (or even hours) is compliant.
⏱️ What this means for uptime: Brief outages (<2 hours) are absorbed by the protocol retry mechanism. Messages buffer at the sender. No data is lost. This is fundamentally different from a real-time API where every second of downtime means failed requests.

Our recommendation: Start with 2 EU regions in active-active (e.g. Frankfurt + Warsaw or Frankfurt + Stockholm). This gives you:

  • Geographic redundancy (different power grids, network backbones, legal jurisdictions within EU)
  • Low latency for both Western and Eastern European customers
  • Compliance with EU data residency (GDPR — data stays in EU)
  • If one region goes down, DNS/health-check routing sends traffic to the other (within seconds)
🧠 Why not just one region with multi-AZ? Multi-AZ within a single cloud region protects against datacenter failures but not against regional outages (which happen — AWS eu-central-1 had major outages in 2023 and 2024). For 99.9% SLA, multi-region is safer and still cost-effective at AP scale.

Load balancer strategy (Nginx or cloud LB)

LayerComponentPurpose
DNS Cloudflare / Route53 GeoDNS + health checks Route senders to the closest healthy region. Automatically remove dead regions.
Regional Nginx or cloud LB (ALB/NLB) TLS termination, rate limiting, AS4 SOAP routing to backend instances.

🔧 Nginx config for AS4: The AS4 endpoint is a single POST /as4/receive SOAP endpoint. Nginx terminates TLS, forwards to upstream Node.js instances. The upstream can be scaled horizontally. Example snippet: 
upstream ap-backend {
    least_conn;
    server 10.0.1.10:3000 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:3000 max_fails=3 fail_timeout=30s;
}

server {
    listen 443 ssl http2;
    server_name ap.mojafaktura.sk;

    location /as4/receive {
        proxy_pass http://ap-backend;
        proxy_read_timeout 120s;
        proxy_http_version 1.1;
    }
}

5. What Does a Pure Peppol AP Core Use a Database For?

This is a great question. The AP Core is not the same as the Service Platform (the customer portal with companies, users, etc.). The AP Core serves as the certified transport layer — and it still needs persistence. Here's exactly what:

Essential (AP certification boundary)

Transactions Required for:
  • Duplicate detection — AS4 step 12: if a message arrives with the same messageId seen before, you must re-send the receipt but not re-process the payload
  • Message replay — if your downstream system crashes, you need to re-deliver the payload
  • Peppol compliance reporting — monthly statistics to the Peppol Authority
  • Audit trail — prove what was sent/received and when

transportProfile, resolvedAt, ttl

What is NOT in the AP Core DB

EntityWhere it livesWhy it's not AP Core
Companies (SMEs)Service Platform PostgreSQLYour customer management — not part of the Peppol transport layer

💡 The clean separation: The AP Core knows about messages but not about customers. It stores:
  • Transaction metadata + payloads (for replay, dedup, compliance)
  • Its own identity (certificates)
  • SMP lookup cache (performance)
It does not store companies, users, invoices, or any business-domain data. That's the Service Platform's job. The AP Core processes messages; the Service Platform processes invoices.

What kind of database?

For a pure AP Core:

  • Transactions: A document DB that supports composite key lookups. Node42 ships with adapters for AWS DynamoDB, Azure CosmosDB, and a local JSON file. DynamoDB is a good fit: you query by messageId (for dedup) or by participantId (for reporting).
  • SMP Cache: Same DB as transactions, or a simple in-memory cache with TTL + periodic refresh. Loss of cache is not catastrophic — just means a DNS lookup penalty on the next send.
  • Certificates: Encrypted at rest in a secrets manager (AWS Secrets Manager, HashiCorp Vault) or in the same DB with envelope encryption.
  • Payloads: Object storage (S3, R2, MinIO, or filesystem) — not the transaction DB. UBL XML payloads can be large (up to 2 GB per Peppol spec), and storing them in DynamoDB would be slow and expensive.
⚡ Prototype shortcut: For early development, use SQLite or a JSON file for the transaction log + in-memory SMP cache + filesystem for payloads. One machine, no cloud dependencies. Migrate to DynamoDB + S3 when you go production.

Assumptions:

  • Peppol B2B mandate drives adoption: ~60,000 SMEs in Slovakia need Peppol connectivity
  • 10% of monthly volume happens on the busiest day (month-end invoicing spike)
  • Busiest hour is ~20% of the busiest day
  • Each transaction is ~1 request (send) + 1 response (receipt) = 2 HTTP round-trips at the AS4 level
  • Additional overhead: SMP lookups (~1 per unique participant), validation

Notes on hardware:

  • Node.js is single-threaded per process. Run 1 instance per CPU core behind Nginx (cluster mode). For a 2-vCPU machine, run 2 Node.js workers.
  • The AS4 processing is mostly I/O bound (network calls for SMP, crypto operations are async, DB writes). CPU is rarely the bottleneck.
  • Memory: Each AS4 payload can be up to 2 GB (Peppol spec). In practice, invoices are 10–100 KB. But you need enough RAM to handle concurrent large payloads during the spike.
  • Disk: Transaction logs and payloads grow. Plan retention: 90 days for operational data, 7 years for audit (Slovak accounting law). Archive old payloads to cold storage.

9. Summary

QuestionAnswer
Is HA required by Peppol? The TIA minimum (98.5%/94%) does not strictly require active-active HA. A DR environment + contingency plan is the minimum.
What do commercial providers do? 99.9% uptime is the market norm — multi-AZ or multi-region, redundant endpoints, often SOC2/ISO27001 certified.
Our recommended architecture? Active-active across 2 EU regions (Frankfurt + Warsaw) behind GeoDNS + Nginx/cloud load balancer.
What DB does a pure AP Core need? Transaction log (dedup + reporting) + certificate store + SMP cache + payload storage. Document DB (DynamoDB) + object storage (S3).
Biggest risk without HA? Not downtime itself, but data loss in the transaction log. The protocol retries cover brief outages, but once a message is accepted, it must be durably stored.
Minimum viable production setup? Single region, 2 app servers behind LB + DynamoDB/S3 + separate DR environment + documented failover plan.
Proto-to-production cost? Starts at ~€15–30/month (single VPS) → ~€400–700/month (2-region HA, 5,000 SMEs).
🎯 Bottom line: The protocol gives you slack (2-hour retry window), but reputation demands 99.9%. Active-active multi-region is the right architecture for a serious AP. Start simple, add regions as you grow. The database question is simpler than it seems: your AP Core needs message-level persistence (transaction log + payloads + certs), not business-level persistence (companies, users, invoices) — that's the Service Platform's job.
