📡 Lesson 3: AS4 Messaging

How the UBL invoice XML travels across the Peppol network — the transport protocol.

From XML to wire

In Lesson 2 we saw the payload: a UBL XML invoice. But how does it actually get from the sender to the receiver? The journey goes through three layers:

┌────────────────────────────────────────────────────────────┐ │ HTTPS (TLS v1.2+) │ │ ┌──────────────────────────────────────────────────────┐ │ │ │ AS4 = SOAP + WS-Security │ │ │ │ ┌────────────────────────────────────────────────┐ │ │ │ │ │ SBDH (Standard Business Document Header) │ │ │ │ │ │ ┌──────────────────────────────────────────┐ │ │ │ │ │ │ │ UBL XML Invoice (payload) │ │ │ │ │ │ │ └──────────────────────────────────────────┘ │ │ │ │ │ └────────────────────────────────────────────────┘ │ │ │ └──────────────────────────────────────────────────────┘ │ └────────────────────────────────────────────────────────────┘

Your Access Point needs to speak all four layers. Fortunately, you don't have to write every byte from scratch — there are existing libraries. But understanding what happens at each layer is essential for debugging.

Phase 1: Discovery (SML + SMP)

Before your AP can send anything, it needs to discover where to send it. Peppol uses two lookup services:

1. SML — Service Metadata Locator

A DNS-based root directory operated by OpenPeppol. Given a participant ID (e.g., 0088:SK4498765432), it returns the URL of that participant's SMP. Think of it as DNS for Peppol — a simple domain name lookup.

Example query pattern:

  Participant ID: 0088:SK4498765432
  → SML query: b-0088-sk4498765432.iso6523-actorid-upis.sml.peppolcentral.org
  → DNS CNAME result: smp.gov.sk

2. SMP — Service Metadata Publisher

A directory operated by the receiving participant's service provider (or the government itself). It answers:

  • What document types does this participant accept? (e.g., "Invoice, CreditNote")
  • Which AP endpoint should the document be sent to? (e.g., https://ap.gov.sk/as4)
  • What's their encryption certificate? (to encrypt the payload for them)
💡 For your MVP: You can skip the SML/SMP discovery and receive documents without it. For sending, you need it to look up where to send. The Node42 library handles this automatically. But you'll also need to register your own customers in your SMP so other APs can find them — more on that later.

Phase 2: Transport — AS4

AS4 (Applicability Statement 4) is a profile of the ebMS 3.0 standard. It defines the exact format for packaging, signing, encrypting, and exchanging business messages over the internet.

Key facts about Peppol AS4:

PropertyValue
TransportHTTPS (TLS v1.2+)
Port443 — both sending and receiving
Message patternOne-Way/Push only — your AP sends, their AP acknowledges. No pull, no sync.
Payload locationMIME attachments (not SOAP body)
AgreementRefurn:fdc:peppol.eu:2017:agreements:tia:ap_provider — fixed value
CompressionOptional (AS4 compression feature)

The SBDH — Routing Envelope

Every Peppol document is wrapped in a Standard Business Document Header (SBDH) before being placed into the AS4 message. This is the routing layer — it tells the receiving AP who sent it, who should get it, and what type of document is inside.

<!-- SBDH — the routing envelope —>
<StandardBusinessDocument
    xmlns="http://www.unece.org/cefact/namespaces/StandardBusinessDocumentHeader">

  <StandardBusinessDocumentHeader>
    <HeaderVersion>1.0</HeaderVersion>

    <!-- Corner 1 (sender, the SME) -->
    <Sender>
      <Identifier Authority="iso6523-actorid-upis">
        0088:SK2023456789
      </Identifier>
    </Sender>

    <!-- Corner 4 (receiver, e.g. government) -->
    <Receiver>
      <Identifier Authority="iso6523-actorid-upis">
        0088:SK4498765432
      </Identifier>
    </Receiver>

    <DocumentIdentification>
      <Standard>urn:oasis:names:specification:ubl:schema:xsd:Invoice-2</Standard>
      <TypeVersion>2.1</TypeVersion>
      <InstanceIdentifier>uuid:3a1b2c3d-...</InstanceIdentifier>
      <Type>Invoice</Type>
      <CreationDateAndTime>2026-06-15T10:30:00+02:00</CreationDateAndTime>
    </DocumentIdentification>

    <BusinessScope>
      <Scope>
        <Type>DOCUMENTID</Type>
        <InstanceIdentifier>
          urn:oasis:names:specification:ubl:schema:xsd:Invoice-2::
          Invoice##urn:cen.eu:en16931:2017#compliant#
          urn:fdc:peppol.eu:2017:poacc:billing:3.0::2.1
        </InstanceIdentifier>
        <Identifier>busdox-docid-qns</Identifier>
      </Scope>
      <Scope>
        <Type>PROCESSID</Type>
        <InstanceIdentifier>urn:fdc:peppol.eu:2017:poacc:billing:01:1.0</InstanceIdentifier>
        <Identifier>cenbii-procid-ubl</Identifier>
      </Scope>
      <Scope>
        <Type>COUNTRY_C1</Type>
        <InstanceIdentifier>SK</InstanceIdentifier>
      </Scope>
    </BusinessScope>
  </StandardBusinessDocumentHeader>

  <!-- The actual UBL XML invoice goes here -->
  <Invoice xmlns="urn:oasis:names:...">
    ...
  </Invoice>

</StandardBusinessDocument>
🔑 Key fields in SBDH:
  • Sender/Receiver: Peppol participant IDs (the end users, not the APs — APs are identified separately in the AS4 header)
  • InstanceIdentifier: A UUID that uniquely identifies this message transmission. Used for tracking and receipt matching.
  • DOCUMENTID scope: The full document type identifier — this tells the receiver exactly which BIS spec the payload conforms to.
  • PROCESSID scope: The business process (billing 01 = standard invoice).

The AS4 Message — MIME Multipart Structure

The AS4 message is a MIME multipart/related message sent over HTTPS. It contains three parts:

Content-Type: multipart/related; boundary="--=_MIME-Boundary"; type="application/xop+xml"

── Part 1: SOAP Envelope (XML) ──
<soap:Envelope xmlns:soap="...">
  <soap:Header>
    <eb:Messaging xmlns:eb="http://docs.oasis-open.org/ebxml-msg/ebms/v3.0/ns/core/200704/">
      <eb:UserMessage>
        <eb:MessageInfo>
          <eb:Timestamp>2026-06-15T10:30:00Z</eb:Timestamp>
          <eb:MessageId>uuid:3a1b2c3d-...@ap.mojafaktura.sk</eb:MessageId>
        </eb:MessageInfo>
        <eb:PartyInfo>
          <!-- Sending AP (Corner 2) -->
          <eb:From>
            <eb:PartyId type="urn:fdc:peppol.eu:2017:identifiers:ap">POP000123</eb:PartyId>
            <eb:Role>http://docs.oasis-open.org/ebxml-msg/ebms/v3.0/ns/core/200704/initiator</eb:Role>
          </eb:From>
          <!-- Receiving AP (Corner 3) -->
          <eb:To>
            <eb:PartyId type="urn:fdc:peppol.eu:2017:identifiers:ap">POP000456</eb:PartyId>
            <eb:Role>http://docs.oasis-open.org/ebxml-msg/ebms/v3.0/ns/core/200704/responder</eb:Role>
          </eb:To>
        </eb:PartyInfo>

        <eb:CollaborationInfo>
          <eb:AgreementRef>
            urn:fdc:peppol.eu:2017:agreements:tia:ap_provider
          </eb:AgreementRef>
          <eb:Service type="cenbii-procid-ubl">
            urn:fdc:peppol.eu:2017:poacc:billing:01:1.0
          </eb:Service>
          <eb:Action>
            busdox-docid-qns::urn:oasis:names:specification:ubl:schema:xsd:Invoice-2::
            Invoice##urn:cen.eu:en16931:2017#compliant#
            urn:fdc:peppol.eu:2017:poacc:billing:3.0::2.1
          </eb:Action>
        </eb:CollaborationInfo>

        <eb:PayloadInfo>
          <eb:PartInfo href="cid:payload@sender">
            <eb:PartProperties>
              <eb:Property name="originalSender">0088:SK2023456789</eb:Property>
              <eb:Property name="finalRecipient">0088:SK4498765432</eb:Property>
            </eb:PartProperties>
          </eb:PartInfo>
        </eb:PayloadInfo>
      </eb:UserMessage>
    </eb:Messaging>

    <!-- WS-Security: digital signature -->
    <wsse:Security xmlns:wsse="...">
      <wsse:BinarySecurityToken
        ValueType="http://docs.oasis-open.org/wss/2004/01/
          oasis-200401-wss-x509-token-profile-1.0#X509v3"
        EncodingType="http://docs.oasis-open.org/wss/2004/01/
          oasis-200401-wss-soap-message-security-1.0#Base64Binary"
        wsu:Id="SenderCert">
        [base64-encoded X.509 certificate of your AP]
      </wsse:BinarySecurityToken>
      <ds:Signature>...</ds:Signature>  <!-- XML-DSig over SOAP body -->
    </wsse:Security>
  </soap:Header>

  <soap:Body>
    <!-- Empty — payload is in the attachment -->
    <xop:Include xmlns:xop="http://www.w3.org/2004/08/xop/include"
                href="cid:payload@sender"/>
  </soap:Body>
</soap:Envelope>

── Part 2: Payload (SBDH + UBL XML) ──
Content-ID: <cid:payload@sender>
Content-Type: application/xml

<StandardBusinessDocument>
  <StandardBusinessDocumentHeader>
    ... (SBDH from previous section) ...
  </StandardBusinessDocumentHeader>
  <Invoice>
    ... (UBL invoice from Lesson 2) ...
  </Invoice>
</StandardBusinessDocument>
💡 The SOAP Body is empty. Peppol requires that all payloads be MIME attachments, not in the SOAP body. The SOAP body just contains an xop:Include reference — like a pointer saying "the real data is over here in the attachment." This is called XOP (XML-binary Optimized Packaging).

Security: Two layers of encryption

LayerCertificatePurpose
TLS (transport)Standard TLS cert (e.g., Let's Encrypt)Encrypts the HTTPS connection between APs. Prevents eavesdropping on the wire. Must match the AP's domain name.
AS4 message securityPeppol PKI certificate (from approved CA)Signs the message (your AP's identity, non-repudiation) and encrypts the payload (only the receiving AP can decrypt). Uses the receiving AP's public key from the SMP.
⚠ Two different certificates! Your AP needs:
  1. TLS certificate for HTTPS — buy from any CA (Let's Encrypt is fine)
  2. Peppol PKI certificate for AS4 signing/encryption — issued through the Peppol Service Desk after accreditation
In the test environment, you can use self-signed for TLS and test PKI certs from OpenPeppol.

The Receipt (MDN)

After the receiving AP successfully processes your message, it sends back a signed receipt (Message Disposition Notification). This is a separate AS4 message that says: "I received message XYZ, the signature verified, the payload is valid." The receipt is itself digitally signed — this provides non-repudiation of receipt.

Your AP must:

  • Send receipts when receiving messages (mandatory)
  • Receive and verify receipts when sending messages (store them as proof of delivery)

What this means for your code

Here's how the sending pipeline maps to functions in your Node.js AP:

Function What it does ────────── ──────────────────────────────── lookupSML(participantID) DNS query → SMP URL querySMP(smpURL, participantID, docType) HTTP GET → endpoint, cert buildSBDH(sender, receiver, docType) XML → SBDH envelope buildSOAP(messageId, from, to, ...) XML → eb:Messaging header signMessage(soapEnvelope, privateKey) XML-DSig → WS-Security encryptPayload(payload, receiverCert) RSA-OAEP + AES-128-GCM buildMIME(soapEnvelope, encryptedPayload) multipart/related POST(url, mimeMessage) HTTPS → receiving AP verifyReceipt(response, truststore) Parse + validate MDN receipt
🔧 Node42 to the rescue: The node42-edelivery library is a pure Node.js toolkit that implements all of the above. No Java, no native dependencies, no external binaries. It handles:
  • SML lookup + SMP discovery
  • AS4 message construction, signing, encryption
  • MIME multipart assembly and HTTP POST
  • Schematron validation
  • Receipt generation and verification
You can use it as a CLI tool or integrate it as a library. This is your head start for the prototype — you don't need to write AS4 from scratch.

Receiving messages

Your AP must also receive AS4 messages. This means running an HTTPS server that:

  1. Listens on port 443 (or behind a reverse proxy like Nginx)
  2. Accepts POST requests with Content-Type multipart/related
  3. Parses the SOAP envelope, extracts the eb:Messaging header
  4. Validates the sender's signature against their Peppol certificate
  5. Decrypts the payload using your private key
  6. Extracts the SBDH + UBL document
  7. Validates the UBL with Schematron
  8. Routes the document to your customer's inbox
  9. Sends back a signed receipt (MDN)

Node42 provides a receiver module too — check their src/receiver/ directory for how to parse incoming AS4 messages.
