🔬 Lesson 12: Simulation Mode — Develop Without Accreditation

How to build and test the full Peppol AP Core without a single PKI certificate, DNS lookup, or network connection.

Why simulation mode exists

Running a certified Peppol Access Point requires:

  • OpenPeppol membership (paid)
  • PKI certificates issued through the Peppol Service Desk
  • Network connectivity to the Peppol SML (DNS)
  • Peppol Testbed accreditation before going live

That's fine — but it means you can't touch the code until you're accredited. Simulation mode solves this. It's a development mode that replaces every external Peppol dependency with an in-memory equivalent:

Real PeppolSimulation mode
SML DNS lookupLocal participant registry (in-memory map)
SMP metadata queryLocal participant registry
AS4 HTTP POST to receiver APDirect function call into receive pipeline
PKI signing (WSS)Fake signature inserted
PKI encryption (AS4)Payload passed through unencrypted
Signed MDN receiptSend path: fake MDN with (simulated-signature) placeholder. Receive path: real RSA-signed MDN via test/fixtures/keys/sim-signing-key.pem — valid XML-DSig structure with DigestValue, but issued by a test key, not a Peppol PKI.
Peppol network errorsConfigurable error injection

The entire AS4 send/receive pipeline is exercised. The only thing fake is the network layer.

💡 This is not a mock. Simulation mode doesn't skip the validation pipeline, the SBDH assembly, the AS4 message structure, or the MDN receipt generation. It runs the real code paths — only the transport layer is replaced.

How it works

The simulation is controlled by the simulationMode flag in src/simulation.js. When set, three things happen:

  1. Send path — sendInvoice() detects simulation mode and calls simulator.simulateSend() instead of making a real AS4 HTTP POST
  2. Receive path — POST /api/simulate/inject calls receiveWebhook() directly with a simulated incoming AS4 message
  3. Participant resolution — lookupParticipant() hits an in-memory registry instead of querying the SML/SMP
Simulation mode data flow 
# Sending an invoice
POST /api/send
  → sendInvoice({ simulationMode: true })
    → simulator.simulateSend()        # fake AS4 POST
      → validateUBL()                  # real Schematron
      → buildSBDH()                    # real SBDH assembly
      → buildAS4Message()              # real message structure
      → simulator.fakeMdnReceipt()    # fake but realistic MDN
    → returns MDN receipt

# Receiving an invoice (simulated inbound)
POST /api/simulate/inject
  → simulator.fakeIncomingMessage()   # build fake AS4 message
    → receiveWebhook()                # real receive pipeline
      → parseAS4Message()              # real parsing
      → validateUBL()                  # real Schematron
      → storeTransaction()             # real DB write
      → simulator.fakeMdnReceipt()     # fake MDN back to sender
    → returns MDN receipt

Enabling simulation mode

The server reads SIMULATION_MODE from the environment. In the dev server, it's set automatically when you use npm run start:sim:

# The npm script sets SIMULATION_MODE=true and NODE_ENV=development
npm run start:sim

# Equivalent to setting these env vars manually:
SIMULATION_MODE=true NODE_ENV=development node server/index.js

To check whether the server is running in simulation mode:

curl http://localhost:3001/api/health
# Returns: { "status": "ok", "simulationMode": true, ... }
⚠ Never set SIMULATION_MODE=true in production. It disables certificate verification and certificate hostname checks. The npm start script does not set it.

The simulated participant registry

When simulation mode starts, it pre-loads a set of known test participants. You can register your own:

# Register a new participant in the simulated network
curl -s -X POST http://localhost:3001/api/simulate/participants \
  -H "Content-Type: application/json" \
  -d '{"id": "9914:SK1122334455", "name": "Test Supplier s.r.o."}'

# List all known participants
curl -s http://localhost:3001/api/simulate/participants | python3 -m json.tool

The registry is in-memory only — it resets when the server restarts. To pre-seed it on every start, edit src/simulator.js and add entries to the participants map.

Simulation via the CLI tool

The server/simulate.js CLI wraps the same simulation operations:

node server/simulate.js --help
# Usage: node simulate.js <command> [options]
#
# Commands:
#   inbound     Simulate receiving an AS4 message from another AP
#   send        Smoke-test sending with auto-generated invoice data
#   register    Add a participant to the simulated network
#   participants  List all participants in the simulated network

Simulate an inbound message from another AP

node server/simulate.js inbound --sender 9914:SK5599887766 --amount 1500
# Builds a fake AS4 message from SK5599887766 to this AP
# Runs it through the full receive pipeline
# Stores the transaction and returns the MDN

Smoke-test sending with auto-generated data

node server/simulate.js send --to 9914:SK4498765432 --amount 2500
# Auto-generates a realistic invoice
# Validates it, builds AS4 message
# Returns a simulated MDN receipt

Register a trading partner

node server/simulate.js register 9914:SK1122334455 --name "Nový dodávateľ s.r.o."

Writing simulation-aware tests

Tests that use simulation mode are fast, deterministic, and require no network. Use enableSimulation() from src/simulation.js to activate it in your test setup:

import { enableSimulation } from '../src/simulation.js';

beforeEach(() => {
  enableSimulation();  // activates simulation mode for all send/receive calls
});

it('validates and sends an invoice in simulation mode', async () => {
  const result = await sendInvoice({ senderId, receiverId, ublXml });
  expect(result.mdn).toBeDefined();
  expect(result.simulation).toBe(true);
});

The regression test suite (test/regression/) uses simulation mode to test the full send/receive cycle against fixture files — no network, no certificates, no flakiness.

Switching from simulation to live

When you have test PKI certificates from OpenPeppol and are ready to run against the real Peppol network:

# 1. Place your test certificates in certs/
#    (the exact path is configured via PEPPOL_CERT_PATH / PEPPOL_KEY_PATH env vars)
ls certs/
# peppol-ap.crt  peppol-ap.key  sml-test.crt

# 2. Set real network mode
export SIMULATION_MODE=false
export NODE_ENV=production

# 3. Start in live mode
npm start

# 4. Run the test suite against live network
npm test
💡 The AP Core is the same code in both modes. Only src/simulation.js changes behaviour. The validation, SBDH, AS4 message building, and receive pipeline are identical. Switching modes doesn't change correctness — it only changes whether the network calls are real or simulated.

Simulation in CI/CD

Run the simulation-based test suite in CI for every pull request — it's fast and requires no secrets:

# .github/workflows/test.yml (conceptual)
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      - run: npm ci
      - run: npm run start:sim &
      - run: npm test
        env:
          SIMULATION_MODE: 'true'

Advanced: Customising the simulator

Adding a pre-seeded participant

Edit src/simulator.js and add to the participants map:

export const participants = new Map([
  ['9914:SK2023456789', { name: 'Pekáreň Pod Hradom s.r.o.', country: 'SK' }],
  ['9914:SK4498765432', { name: 'Mesto Trnava', country: 'SK' }],
  ['9914:SK5599887766', { name: 'Test AP Slovakia', country: 'SK' }],
  // Add your own participants here
  ['9914:SK1122334455', { name: 'Your Supplier s.r.o.', country: 'SK' }],
])

Simulating error responses

For testing retry logic and error handling, you can configure the simulator to return specific error responses:

# Set via environment variable — useful for chaos testing
SIMULATION_ERROR=certificate_expired npm run start:sim

The available error codes depend on what's implemented in src/simulator.js — check the SIMULATION_ERROR_CODES map there.

Injecting a custom AS4 message

For advanced testing, you can inject a manually crafted AS4 message directly into the receive pipeline via POST /api/simulate/inject with the rawAS4 option:

curl -s -X POST http://localhost:3001/api/simulate/inject \
  -H "Content-Type: application/json" \
  -d '{
    "rawAS4": "BASE64_ENCODED_AS4_MESSAGE",
    "senderId": "9914:SK5599887766"
  }'
