✅ AP Core Validation Rules

What the AP Core checks when you call /api/send or /api/validate.

💡 Two validation layers:
  • XSD — XML Schema: checks structure, cardinality, data types (mechanical, done by the XML parser)
  • Schematron — Business rules: cross-field logic, code lists, conditional requirements (done by Node42's bundled Schematron engine)
The AP Core runs both before accepting any document for transport. This reference documents the Schematron rules in human-readable form.

1. Header fields (mandatory)

These are the absolute minimum fields required by Peppol BIS Billing 3.0. If any of these is missing or invalid, the AP Core rejects the document with PEPPOL-EN16931-R001 or similar.

BT#FieldUBL ElementRule
— CustomizationID /Invoice/cbc:CustomizationID Must be exactly urn:cen.eu:en16931:2017#compliant#urn:fdc:peppol.eu:2017:poacc:billing:3.0. Any other value → rejection.
— ProfileID /Invoice/cbc:ProfileID Must be exactly urn:fdc:peppol.eu:2017:poacc:billing:01:1.0 for basic billing.
BT-1 Invoice number /Invoice/cbc:ID Must be present and unique per seller. Free text, max 100 chars typically.
BT-2 Issue date /Invoice/cbc:IssueDate Format YYYY-MM-DD. Must be a real calendar date.
BT-3 Invoice type code /Invoice/cbc:InvoiceTypeCode Allowed values: 380 (invoice), 381 (credit note), 383 (corrected), 384 (self-billed), 386 (prepayment), 389 (self-billed credit note).
BT-5 Currency /Invoice/cbc:DocumentCurrencyCode ISO 4217 code. Only one currency per invoice. Slovak defaults: EUR.
BT-34 Seller endpoint ID /Invoice/cac:AccountingSupplierParty/cac:Party/cbc:EndpointID Must be a valid Peppol participant ID with @schemeID. For Slovakia: 9914:SK2023456789.
BT-46 Buyer endpoint ID /Invoice/cac:AccountingCustomerParty/cac:Party/cbc:EndpointID Same format as seller endpoint. Must resolve to an existing Peppol participant.
BT-31 Seller VAT ID //cac:PartyTaxScheme[cac:TaxScheme/cbc:ID='VAT']/cbc:CompanyID Must match the seller's country VAT format. For SK: SK2023456789 (SK + 10 digits).
BT-44 Buyer name //cac:AccountingCustomerParty//cac:PartyLegalEntity/cbc:RegistrationName Required. The full formal name.
⚠ AP Core enforces: If any mandatory (1..1) field is missing, the AP Core returns validation_failed with the XPath of the missing element before attempting any Peppol transport. The invoice never reaches the network.

3. Monetary totals — cross-field validation

These are classic Schematron rules. The AP Core's Schematron engine checks every mathematical relationship:

Rule IDCheckWhat fails it
PEPPOL-EN16931-R029 BT-112 (TaxInclusiveAmount) = BT-109 (TaxExclusiveAmount) + BT-110 (VAT total) A rounding error or omitted VAT subtotal. Difference > €0.01 → rejection.
PEPPOL-EN16931-R030 BT-115 (PayableAmount) = BT-112 (TaxInclusive) − BT-113 (Prepaid) + BT-114 (Rounding) Prepaid amount deducted but not reflected in PayableAmount.
PEPPOL-EN16931-R031 BT-106 (LineExtensionAmount) = Σ of all BT-131 (InvoiceLine net amounts) Line items don't add up to the header total.
PEPPOL-EN16931-R032 BT-109 (TaxExclusive) = BT-106 − BT-107 (Allowances) + BT-108 (Charges) Document-level allowances/charges not reflected in the total.
PEPPOL-EN16931-R033 BT-110 (VAT total) = Σ of all BT-117 (TaxSubtotal/TaxAmount) VAT breakdown mismatch.
PEPPOL-EN16931-R034 Each BT-131 (line net) = BT-129 (quantity) × BT-146 (unit price) ± line allowances/charges Line item arithmetic error.
💰 Practical tip: These cross-field checks catch most real-world invoice errors. If an SME types €1,200 as the line total but has 10 units at €100, the AP Core catches it. This is why SMEs will pay you — you prevent them from sending invalid invoices that their customer's AP would reject anyway.

7. Document type identifier (the long string)

One of the most common validation failures. The SBDH DOCUMENTID scope and the AS4 Action field must contain the exact document type identifier for Peppol BIS Billing 3.0:

urn:oasis:names:specification:ubl:schema:xsd:Invoice-2::
Invoice##urn:cen.eu:en16931:2017#compliant#
urn:fdc:peppol.eu:2017:poacc:billing:3.0::2.1

For credit notes, the ::Invoice## part is replaced by ::CreditNote## (but the namespace stays CreditNote-2). The AP Core constructs this from the document type in the UBL — if the platform sends it incorrectly, the discovery (SMP lookup) will fail because the receiver's SMP won't recognise the document type.

📁 Where these rules live in Node42: The library bundles pre-compiled Schematron files at src/assets/schematrons/billing/:
  • cen-en16931-ubl.sef.json — CEN rules (European standard)
  • peppol-en16931-ubl.sef.json — Peppol-specific rules
These are loaded at startup and applied to every document in /api/send and /api/validate. You can add Slovak PASR rules as a third Schematron file.

What this means for the Service Platform

When the platform calls /api/send, the AP Core validates against all of the above before touching the network. The platform does not need to implement these rules itself — but if it wants to catch errors before the SME clicks "Send", it can call /api/validate first, which runs the exact same validations.

The validation error response looks like:

{
  "error": "validation_failed",
  "details": [
    {
      "rule": "PEPPOL-EN16931-R029",
      "severity": "fatal",
      "message": "TaxInclusiveAmount (1656.25) must equal TaxExclusiveAmount (1325.00) + VAT total (330.00)",
      "location": "/Invoice/cac:LegalMonetaryTotal/cbc:TaxInclusiveAmount"
    }
  ]
}
🔌 Bottom line: The AP Core's validation contract is as important as its transport contract. The Platform knows: "I send UBL XML → the AP Core either sends it or tells me exactly why it can't." The validation rules above are not optional — they're what OpenPeppol requires every AP to enforce, and they're what the Peppol Testbed tests during accreditation.
