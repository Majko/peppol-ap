🔌 AP Core Interface Specification

The contract between the Peppol AP Core and any Service Platform.

💡 This is the interface boundary. Everything on this page belongs to the AP Core side. The Service Platform (your React portal + Express + PostgreSQL) calls these operations. An accounting software vendor could also call them directly via the REST API. The AP Core never touches user management, databases, or business logic.

Operation 1: Send Invoice

POST /api/send

Push a UBL invoice to a receiver on the Peppol network. The AP Core handles the entire AS4 pipeline: SML/SMP discovery → build SBDH → construct AS4 SOAP → sign → encrypt → send → verify receipt.

Request

FieldTypeCardDescription
sender_idstringREQYour customer's Peppol participant ID, e.g. 9914:SK2023456789
receiver_idstringREQThe receiver's Peppol participant ID, e.g. 0088:SK4498765432
ubl_xmlstringREQThe full UBL Invoice or CreditNote XML (Peppol BIS Billing 3.0)
document_typestringOPTIf omitted, derived from the UBL document. Explicit: invoice or credit_note
process_idstringOPTIf omitted, uses default billing process. Explicit: urn:fdc:peppol.eu:2017:poacc:billing:01:1.0

Response (success)

FieldTypeDescription
message_idstringPeppol message ID (uuid:...@ap.mojafaktura.sk). Used for tracking and receipts.
statusstringOne of: sent (AP accepted), delivered (receipt received from remote AP)
receiptstring | nullThe signed MDN receipt XML, if the remote AP acknowledged
timestampstringISO 8601 timestamp of completion

Response (error)

FieldTypeDescription
errorstringError code: validation_failed, lookup_failed, send_failed, receipt_timeout
detailsarrayList of error details. For validation errors: Schematron violation messages.

Example

POST /api/send
Content-Type: application/json

{
  "sender_id": "9914:SK2023456789",
  "receiver_id": "0088:SK4498765432",
  "ubl_xml": "<?xml version=\"1.0\"...><Invoice xmlns=...>..."
}

→ 200 OK

{
  "message_id": "uuid:a1b2c3d4-...@ap.mojafaktura.sk",
  "status": "delivered",
  "receipt": "<?...><SignalMessage>...</SignalMessage>",
  "timestamp": "2026-07-03T14:30:00Z"
}
⚙ What happens inside the AP Core during send:
  1. Validate the UBL XML against Schematron rules → reject with validation_failed if invalid
  2. Resolve receiver_id via SML (DNS) → find their SMP → get endpoint URL + encryption certificate
  3. Wrap UBL in SBDH envelope (sender = sender_id, receiver = receiver_id)
  4. Build AS4 SOAP header (eb:Messaging with AgreementRef, Service, Action, PartyInfo)
  5. Sign the SOAP envelope with your AP's Peppol PKI signing certificate
  6. Encrypt the SBDH+UBL payload with the receiver's encryption certificate
  7. Assemble MIME multipart/related message
  8. POST to the receiver's endpoint over HTTPS (port 443)
  9. Wait for receipt (MDN) from the receiver — verify their signature
  10. Return result to the platform

Operation 2: Receive Invoice (Webhook)

POST → [your registered callback URL]

When the AP Core receives an AS4 message from another AP on port 443, it calls your registered webhook with the extracted document. The AP Core automatically generates and sends the signed MDN receipt back to the sending AP.

Webhook registration

Tell the AP Core where to deliver received documents:

EndpointMethodBody
POST /api/webhook/register POST { "url": "https://app.mojafaktura.sk/api/webhook/invoice-received", "secret": "whsec_..." }

Webhook payload (AP Core → Platform)

FieldTypeDescription
eventstringinvoice.received
message_idstringPeppol message ID from the sending AP
sender_idstringThe C1 sender's Peppol participant ID (from SBDH)
receiver_idstringThe C4 receiver's Peppol participant ID (your customer)
ubl_xmlstringThe extracted UBL XML document
document_typestringinvoice or credit_note
received_atstringISO 8601 timestamp

Webhook signature

Each webhook call includes a X-Signature-256 header: HMAC-SHA256 of the JSON body, keyed with the shared secret. The platform verifies this to confirm the webhook came from the AP Core.

Example

POST https://app.mojafaktura.sk/api/webhook/invoice-received
Content-Type: application/json
X-Signature-256: a1b2c3d4e5f6...

{
  "event": "invoice.received",
  "message_id": "uuid:x9y8z7w6-...@ap.druhafirma.sk",
  "sender_id": "9914:SK5599887766",
  "receiver_id": "9914:SK2023456789",
  "ubl_xml": "<?xml...><Invoice xmlns=...>...",
  "document_type": "invoice",
  "received_at": "2026-07-03T15:00:00Z"
}
⚙ What happens inside the AP Core on receive:
  1. Listen on port 443 for incoming POST with Content-Type multipart/related
  2. Parse the SOAP envelope → extract eb:Messaging header
  3. Verify the sender's digital signature against their Peppol certificate (fetched from SMP)
  4. Decrypt the payload using your AP's private key
  5. Parse SBDH → extract sender (C1), receiver (C4), document type
  6. Validate UBL with Schematron
  7. Generate and sign MDN receipt → send back to the sending AP
  8. Call your registered webhook with the extracted data
  9. If webhook fails, retry 3 times with backoff, then store for manual replay

Operation 3: Get Status

GET /api/status/{message_id}

Check the delivery status of a previously sent message.

Response

FieldTypeDescription
message_idstringThe message ID from the send response
statusstringpending | sent | delivered | failed
receiptstring | nullThe MDN receipt XML if delivered
errorstring | nullError detail if failed
retriesnumberNumber of send attempts
updated_atstringISO 8601 of last status change

Operation 4: Lookup Participant

GET /api/lookup/{participant_id}

Resolve a Peppol participant ID to its SMP metadata — useful for checking if a participant exists before sending, or viewing their capabilities.

Response

FieldTypeDescription
participant_idstringThe queried participant ID
smp_urlstringThe SMP endpoint for this participant
servicesarrayList of supported document types and their endpoints
services[].document_typestringe.g. urn:oasis:names:specification:ubl:schema:xsd:Invoice-2::Invoice##...
services[].process_idstringe.g. urn:fdc:peppol.eu:2017:poacc:billing:01:1.0
services[].endpointstringThe AP endpoint URL, e.g. https://ap.gov.sk/as4
services[].certificatestringThe receiving AP's encryption certificate (PEM)
resolved_atstringISO 8601 timestamp
💡 SMP response caching. SMP metadata changes infrequently. The AP Core caches results for 1 hour to avoid redundant DNS lookups. Forced refresh with ?refresh=true.

Operation 5: Validate Document

POST /api/validate

Validate a UBL document against Peppol Schematron rules before sending. The platform can call this to catch errors before the SME clicks "Send."

→ See AP Core Validation Rules for the complete list of what is checked.

Request

FieldTypeCardDescription
ubl_xmlstringREQThe UBL XML to validate
schemastringOPTWhich ruleset: peppol (default), cen (EN 16931 only), both

Response

FieldTypeDescription
validbooleantrue if all rules pass
errorsarrayList of violations (empty if valid)
errors[].rulestringSchematron rule ID, e.g. PEPPOL-EN16931-R001
errors[].messagestringHuman-readable description
errors[].locationstringXPath to the offending element
errors[].severitystringfatal | warning
warningsarrayNon-blocking recommendations

What this enables

With this interface, the Service Platform is completely decoupled from the Peppol transport layer. Several architectural possibilities open up:

  • Your React portal calls /api/send when an SME clicks "Send"
  • An accounting software calls /api/send directly (via the integration API, which proxies to the AP Core)
  • A different company could build their own portal on top of your AP Core, paying you for transport only
  • The AP Core can be replaced with a different AS4 implementation (e.g., Java-based Phase4) without changing the platform code — as long as the interface stays the same
  • Single-server prototype: the AP Core functions are just in-process calls. Production: they become HTTP calls to the separate AP Core server. The interface contract is identical.
🎯 Prototyping tip: Start with the AP Core as in-process JavaScript functions (just import Node42 and call it). The Service Platform calls the same interface. When you split servers in week 4, you wrap the same functions behind Express routes — no logic changes, just adding HTTP.
