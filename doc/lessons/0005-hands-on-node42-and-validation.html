🛠️ Lesson 5: Hands-on — Node42, Validation & Network Probing

Installing the tools, validating real invoices, and talking to the live Peppol network — all without accreditation.

What you'll accomplish

By the end of this session, you will have:

  1. Node42 installed and configured on this server
  2. A sample Peppol invoice validated against real Schematron rules
  3. Probed the live Peppol network to resolve real participants

💡 No accreditation needed. Everything below works with public information and local tools. The only thing you can't do yet is send an AS4 message — that requires test PKI certificates from OpenPeppol (which comes after membership). But you can prepare every other layer.

Step 1: Install Node42

Node42 is a pure Node.js toolkit — no Java, no native dependencies. Install it globally:

npm install -g @n42/edelivery

Verify it works:

n42-edelivery --version
→ 0.2.85

n42-edelivery --help
→ Shows commands: init, pki, send, validate, report
⚠ Run this on this server now. The peppoll-access directory is your working space. You can install Node42 globally and use it from anywhere.

Step 2: Initialize your workspace

Node42 needs a workspace with certificates, Schematron rules, and a document template. Let's create one:

cd ~/projects-dir/ideas/pepoll-access
n42-edelivery init

This creates .node42/ in your working directory with:

.node42/
├── certs/
│   ├── cert.pem          ← your AP certificate (placeholder)
│   ├── key.pem           ← your private key (placeholder)
│   └── truststore.pem    ← Peppol root CA certificates
└── templates/
    └── ubl.json          ← UBL document descriptor

The init command creates placeholder certificates and fetches the latest Peppol truststore and Schematron rules. For now, we'll use the test certificates that come with the installation.

But the real value is seeing the output format: which Schematron rules were checked, the rule IDs, and how violations are reported. Try intentionally breaking something (e.g., change the VAT rate to 15%) and run validation again to see the error format.

Step 4: Probe the live Peppol network

You can query real Peppol participants right now. The OpenPeppol SMP Lookup is a public service. Let's use Node42's built-in SMP resolution:

# Resolve a participant — try the Netherlands Peppol Authority test OIN
n42-edelivery send peppol \
  --env test \
  --sender-id "iso6523-actorid-upis::9914:SK2023456789" \
  --receiver-id "iso6523-actorid-upis::0000:00000000000000000000" \
  --document /tmp/test-invoice.xml \
  --dryrun

The --dryrun flag is crucial — it performs the SML/SMP lookup and constructs the AS4 message but does not send it. This lets you see:

  • Whether the receiver exists and what documents they accept
  • The endpoint URL of their AP
  • Their encryption certificate (needed for the AS4 encryption)
  • The full AS4 message structure that would be sent

You can also probe the network using the public web tools:

Next steps toward sending

To actually send an AS4 message, you need test PKI certificates. The path is:

  1. Join OpenPeppol as a member (fee-based, starts at ~€2,000/year for small companies)
  2. Sign the Service Provider Agreement with the Slovak Peppol Authority (Finančné riaditeľstvo SR)
  3. Request test PKI certificates through the Peppol Service Desk
  4. Run the Peppol Testbed — automated tests that verify your AP sends/receives correctly
  5. Request production certificates → go live

But you don't need to wait. You can build and test everything except the actual AS4 network call right now. The validation, SMP probing, webhook receiver, and AP Core interface are all fully developable today. When the certs arrive, you just plug them in and send.
