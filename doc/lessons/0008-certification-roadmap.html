1. The Big Picture: 3 Parallel Workstreams

The accreditation process is not a single sequence. It's 3 parallel workstreams that converge at specific points. Understanding this is the key to minimising the timeline.

╔══════════════════════════════════════════════════════════════╗ ║ CERTIFICATION TIMELINE — 3 PARALLEL STREAMS ║ ╚══════════════════════════════════════════════════════════════╝ STREAM A: NATIONAL ACCREDITATION (PA SK) ┌─────────────────────────────────────────────────────────┐ │ Prepare docs ─→ Submit application ─→ 30-day evaluation │ │ ─→ Sign SPA │ │ ─→ Receive cert │ └─────────────────────────────────────────────────────────┘ STREAM B: OPENPEPPOL MEMBERSHIP ┌─────────────────────────────────────────────────────────┐ │ Apply ─→ Pay fees ─→ Membership approved │ └─────────────────────────────────────────────────────────┘ │ ▼ [Both needed for Test PKI Cert] │ ▼ STREAM C: TECHNICAL (AP CORE + TESTBED) ┌─────────────────────────────────────────────────────────┐ │ AP Core ready ─→ Test PKI cert ─→ Testbed (6 tests) │ │ ─→ Testbed report │ └─────────────────────────────────────────────────────────┘ │ ▼ [Testbed report submitted to PA SK → Evaluation continues] │ ▼ CONVERGENCE: Sign SPA → Accreditation cert → Production PKI → SMP Access

Key insight: Streams A and B can start immediately (legal documents + membership application). Stream C is mostly ready (we have the AP Core) but needs the test PKI cert, which requires Streams A and B to be in progress first.

5. Peppol Testbed — What We Need to Pass

The Testbed consists of 6 mandatory eDelivery test cases that verify our AP's ability to send and receive valid Peppol BIS documents. Based on the OpenPeppol Testbed documentation:

#Test caseWhat it verifiesOur readiness
1 Message submission AP can construct a valid AS4 message with SBDH envelope, sign it, and submit it to a receiving AP endpoint. ✅ We have as4/message.js + as4/sbdh.js working
2 Message reception AP can receive an incoming AS4 message, verify signature, decrypt payload, extract SBDH and UBL document. 🟡 Reception is built but needs end-to-end test with real certs
3 Message-level response (MDN) AP generates a valid signed MDN Receipt (nonce) and sends it back to the sender. ✅ Simulator generates MDN receipts. Node42 handles this natively.
4 Payload validation AP validates the UBL payload against Schematron rules before accepting it. ✅ Validator expanded: 21 rules implemented (Phase 22). Covers type codes 383/386, R008, R020, R032, R042, R050, R055. Still to do: ~40–60 more BIS 3.0 rules + SK PASR rules.
5 Participant discovery (SMP lookup) AP can resolve a participant ID via SML (DNS NAPTR) → SMP (HTTP XML) → obtain endpoint URL and certificate. 🟡 Node42 has lookupParticipant(). We wrap it with a fallback. Needs production verification.
6 Error handling AP correctly handles error scenarios: invalid participant, malformed message, expired cert, etc. 🟡 Error handling exists but needs hardening for all error codes.

Additionally, the Slovak-specific testing (Slovakia Testbed Report) covers:

  • Correct handling of Slovak Peppol BIS Billing 3.0 SK extension (national VAT IDs, specific code lists)
  • IS EFA reporting format compliance (for the 5th corner — tax authority reporting)
  • Integration with the centralised Slovak SMP (once PASR points 4 and 6 become mandatory — check current status)
📋 Slovak PASR — implemented (Phase 22):
  • PASR-SK-001: scheme 0245 endpoints must be exactly 10 digits
📋 Slovak PASR — waiting for spec:
  • Tax Data Document — "mandatory Tax Data Document will be shared in due time" (PASR Section 6)
  • IS EFA / 5th-corner reporting format details
  • Additional VAT rules from Slovak VAT Act (Zákon č. 222/2004 Z. z.)
🔧 Technical gaps to close before Testbed:
  1. Remove dryrun: true in src/as4/node42.js — done in Slice 3 (verify)
  2. Wire certificate loading from identity store (Slice 3)
  3. Implement and test the full AS4 receive endpoint (currently relies on Node42's receiver)
  4. Expand UBL validator to cover all Peppol BIS Billing 3.0 rules (Phase 22 added 6 rules — more remain)
  5. Add Slovakia-specific PASR validation rules (SK VAT ID format, national extensions)
  6. Implement proper error responses for all Testbed error scenarios

Estimated effort: 2–4 weeks of focused work, running in parallel with Streams A and B.

8. SPA Operational Obligations — What You're Signing Up For

The Peppol Service Provider Agreement (SPA) v4.0.2 is the contract we will sign with PA SK upon accreditation. It contains binding operational obligations that directly affect our AP implementation. These are not optional — they are contractual requirements.

SPA ClauseRequirementImpact on our APPriority
9.4.2 — Logging Log all sending and receiving of business documents. Logs must be kept for at least 3 months (or longer if required by law). Must provide log access to other APs and PA SK on reasonable request. Our in-memory Map transactions store is insufficient. We MUST implement persistent storage (Slice 2 — SQLite adapter) with minimum 90-day retention before accreditation. Critical
9.4.3 — Backup & Continuity Must have backup, continuity and recovery procedures in place for all Peppol Services. Active-active multi-region architecture (Lesson 6) and documented DR plan. Database backups with tested recovery. This is a contractual requirement, not just good practice. Critical
9.4.9 — Capacity & Scaling Must have sufficient resources. If minimum service levels cannot be met due to insufficient capacity, must scale systems appropriately. Our capacity planning model (Lesson 6) must be documented and tracked. Auto-scaling or ability to provision additional capacity within SLA window. Critical
9.4.1 — Testing Must perform testing to ensure service offerings comply with Peppol Interoperability Framework and PA-specific requirements. Testbed is just the start. We need ongoing regression testing whenever we update AP software. This should be automated. High
9.4.4 — Alerts & Migration Must monitor OpenPeppol alerts, warnings, and hot-fixes. Must adhere to published migration plans and mandated dates. We need an operational process to track OpenPeppol communications. Missing a PKI migration deadline means losing network access. High
9.4.5 — Support Services Must provide: setup/support to end users, published contact email + phone for incident reporting, incident response, subscription to service messages. We need a support email address, phone number, and a status page before going live. Documented response times in our SLA. High
9.4.8 — Reporting (EUSR + TSR) Must submit monthly End User Statistics Report and Transaction Statistics Report to OpenPeppol in their defined format. Our transaction store must support aggregation queries for monthly reporting. Tickstar offers this as add-on. For our own AP we need a reporting module. High
9.5.3 — Document Validation Must ensure datasets sent on behalf of end users are technically correct and valid according to the relevant Peppol Dataset Type rules. Phase 22 expanded the validator from 15 to 21 rules (R008, R020, R032, R042, R050, R055). Remaining: ~40–60 more BIS 3.0 rules and SK PASR rules. This is an ongoing effort. 🟡 Partially done — ongoing
10.3 / 17.1 — Security Must implement appropriate technical and organisational measures to protect integrity and continuous operation. Protect against accidental loss, alteration, unauthorised access. HMAC webhook signing (Issue 007), encrypted cert storage, access controls, security documentation. Must provide security documentation to PA SK on request. High
10.5 — Breach Notification Must notify PA SK without delay about data breaches, disturbances, or errors that may endanger the Peppol Network. We need an incident response plan with defined notification procedures. Breach notification is a contractual obligation. High
22.2 — Termination Either party may terminate with 6 months written notice. We must plan for this in our business model. If we cease AP operations we have 6 months to transition all participants. Medium
16 — Data Ownership End users retain all IP rights to their data. AP must not claim ownership. Data cannot be shared with third parties except as required for Peppol operation or as agreed with end user. Our terms of service must clearly state this. We must ensure our data processing agreements reflect this. Medium
19.3 — Liability Cap Liability capped at €500,000 per event, €1,000,000 annual maximum. No liability for indirect damages (except breach of confidentiality). Our customer contracts should reference this limit. We should not accept higher liability than we have from OpenPeppol. Medium
📋 Key takeaway: The SPA converts what might seem like "good engineering practice" into binding contractual obligations. The persistent transaction store (Slice 2), multi-region DR architecture (Lesson 6), HMAC webhook signing (Issue 007), and expanded UBL validator are not optional features — they are requirements for accreditation. Every item in "Critical" and "High" priority rows above should be completed before we sign the SPA.

10. Fallback Options If We Don't Make the Deadline

Even in the worst case (certification delayed past January 2027), there are options:

Option 1: White-label reseller bridge (recommended fallback)

Partner with an existing certified provider like Tickstar, ecosio, or PeppolEDGE. You get:

  • Your own Peppol certificate under their infrastructure
  • Immediate Peppol connectivity — no need to wait for your own certification
  • You appear as a certified provider in the OpenPeppol directory (under their SMP)
  • When your own cert arrives, migrate participants seamlessly (same SMP ID, same endpoint)

Cost: ~€250–€1,000/mo depending on volume. Tickstar explicitly advertises this model: "You get your own certificate through your membership in OpenPeppol, but the SMP and Access Point infrastructure, maintenance and hosting are all provided by us."

Option 2: Apply as an existing provider (accelerated path)

If we first certify with another Peppol Authority (e.g., OpenPeppol directly if no national authority exists), then apply to PA SK as an existing accredited provider. The process for existing providers is shorter (steps 1→2→3→4→5→6→7 with no test PKI cert step) — estimated 6–8 weeks instead of 10–14.

Option 3: Partnership with a Slovak certified provider

Partner with one of the 36 already-certified Slovak CPDS providers. They handle the Peppol transport; you handle the customer-facing API and portal. This is essentially a revenue-share model. Lower margin, but zero certification risk.
